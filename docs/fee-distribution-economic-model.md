# Fee Distribution Contract — Economic Model

## Overview

The `FeeDistributionContract` collects protocol fees generated by Bridge Watch
services (bridge monitoring, price aggregation, health scoring) and distributes
them to three classes of stakeholders according to configurable allocation
ratios.

```
Protocol fees
     │
     ▼
FeeDistributionContract
     │
     ├── 50 % ──► Staker pool   (distributed pro-rata to token stakers)
     ├── 30 % ──► Governance pool (vested linearly to governance participants)
     └── 20 % ──► Treasury       (swept directly on each distribution)
```

Ratios are expressed in basis points and may be updated by the admin at any
time.  All three values must sum to 10 000 (100 %).

---

## Stakeholder Roles

| Role | How they participate | How they earn |
|---|---|---|
| **Stakers** | Lock the designated staking token | Pro-rata share of the staker pool on each distribution |
| **Governance participants** | Assigned vesting schedules by the admin | Linear unlock over a configured vesting window |
| **Treasury** | Fixed address set at initialisation | Direct transfer of treasury slice on every distribution |
| **Fee collectors** | Protocol contracts authorised by admin | Deposit fees; do not earn directly |

---

## Fair-Share Algorithm

The contract uses the **dividend-per-share** (DPS) pattern to allocate staker
rewards fairly without iterating over all stakers on every distribution.

### State

| Variable | Storage | Description |
|---|---|---|
| `total_staked` | Instance | Sum of all current stake weights |
| `acc_fee_per_share` | Per fee token (Persistent) | Monotonically increasing accumulator, scaled by `PRECISION = 1e12` |
| `StakerDebt(staker, token)` | Persistent | Debt checkpoint at last weight change |
| `StakerPending(staker, token)` | Persistent | Harvested but unclaimed rewards |

### Distribution step

When fees for token `T` are distributed to the staker pool:

```
Δ = stakers_amount × PRECISION / total_staked
acc_fee_per_share[T] += Δ
```

### Pending reward query (view)

```
accrued  = stake × acc_fee_per_share[T] / PRECISION
new_rewards = accrued − reward_debt[T]
pending  = stored_pending[T] + new_rewards
```

### Stake-weight change protocol

Before any change to a staker's weight (stake, unstake, or compound), the
contract calls `harvest_rewards` which moves `new_rewards` into
`StakerPending` and advances `StakerDebt` to `accrued`.  After the weight
change, `sync_debts` resets `StakerDebt = new_stake × acc_fee_per_share /
PRECISION`.

This guarantees:

- **Late-staker exclusion** — a staker joining after a distribution has
  `reward_debt` equal to their full hypothetical entitlement, so their
  retroactive reward is zero.
- **No unbounded iteration** — gas cost per operation is O(number of fee
  tokens), not O(number of stakers).
- **No rounding drift** — the treasury receives the arithmetic remainder after
  applying the staker and governance basis-point shares, so dust always
  reaches a real address.

### Orphan staker fees

When no tokens are staked (`total_staked = 0`) during a distribution, the
staker allocation would otherwise be lost.  The contract routes this amount
to the `governance_pool` so no fees are permanently burned.

## Vesting (Governance Pool)

The admin creates vesting schedules by drawing from the per-token
`governance_pool` bucket.

### Schedule parameters

| Parameter | Description |
|---|---|
| `beneficiary` | Address that claims the vested tokens |
| `token` | Fee token being vested |
| `total_amount` | Tokens to vest (must not exceed `governance_pool` balance) |
| `duration_secs` | Total vesting window in seconds |
| `cliff_secs` | Minimum wait before any tokens can be claimed |

### Linear unlock formula

```
elapsed      = now − start_time
vested_total = total_amount × min(elapsed, duration) / duration
claimable    = vested_total − claimed_amount
```

The cliff blocks any claim until `elapsed ≥ cliff`.  After the cliff the full
linear function applies, so tokens that accrued during the cliff are claimable
immediately after it passes.

## Auto-Compounding

Stakers may opt into auto-compounding at stake time or via a follow-up stake
call.  When enabled:

- `claim_fees` re-adds the pending reward to the staker's **stake weight**
  rather than transferring tokens out.
- `compound_rewards` is a keeper-friendly variant callable by any address
  (no staker signature required) when compound mode is active.

The effective result is that compounding stakers see their share of future
fee rounds increase without needing to manually restake.

> **Note** — compounding increases the staker's *weight* (`StakerAmount`).
> The underlying reward tokens remain in the contract.  When the staker
> eventually calls `unstake` they recover only their original staking-token
> deposit; compounded rewards are realised through a higher share of
> subsequent fee rounds (and must be claimed explicitly as fee rewards).

---

## Batch & Automated Distribution

`distribute_fees` accepts an optional list of token addresses.  Passing an
empty `Vec` processes every registered fee token in a single transaction.

The admin may set a `distribution_interval` (seconds).  If the interval is
non-zero, `collect_fees` automatically triggers a distribution whenever the
interval has elapsed since the last one, converting fee collection into an
implicit scheduled event.

---

## Emergency Mechanisms

The contract includes two admin-controlled safety nets:

1. **`set_emergency(true)`** — activates an emergency halt that blocks
   `collect_fees`, `distribute_fees`, `stake_for_fees`, and `claim_fees`.
   Read operations (`get_*`) and `unstake` remain available.

2. **`emergency_withdraw(token, recipient, amount)`** — transfers an
   arbitrary amount of any token out of the contract.  Only callable when
   emergency mode is active.

Emergency mode can be deactivated by the admin at any time to resume normal
operation.

## Gas Considerations

| Operation | Complexity |
|---|---|
| `collect_fees` | O(1) per token |
| `distribute_fees(n tokens)` | O(n) |
| `stake_for_fees` / `unstake` | O(n fee tokens) — harvest + sync debts |
| `claim_fees` | O(n fee tokens) — harvest |
| `compound_rewards` | O(n fee tokens) |
| `claim_vested` | O(1) |

The dominant cost is the per-fee-token harvest/sync loop in staking
operations.  Keeping the registered fee-token list small (recommended: ≤ 20)
ensures acceptable gas costs.

---

## Security Properties

- **Reentrancy** — Soroban's execution model is synchronous and single-call;
  reentrancy is not possible at the VM level.
- **Integer overflow** — all arithmetic uses checked operations with explicit
  `expect("overflow")` panics.
- **Access control** — privileged functions require `admin.require_auth()`.
  Fee collection requires either admin auth or explicit collector registration.
- **Ratio invariant** — `validate_ratios` is called on every update; the
  contract cannot enter a state where ratios do not sum to 10 000.
- **Vesting solvency** — `create_vesting_schedule` checks that
  `governance_pool ≥ amount` before committing, preventing over-allocation.
